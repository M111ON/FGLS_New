 ลำดับจริงในระบบตอนนี้:

  1. เริ่มระบบ (startup)

  - สร้าง LUT/route table ก่อนใช้งาน เช่น geo_addr_net_init()
  - จุดนี้คือเตรียม world mapping (AB/face/spoke/polarity) ให้ O(1)
  - อ้างอิง: src/tgw_dispatch_v2.h, src/geo_addr_net.h

  2. ตอน packet/addr วิ่งเข้า dispatch (runtime per event)

  - geo_net_encode(addr) แตก address เป็น geometry fields
  - จากนั้น TGW ใช้ผลนี้ไปตัดสิน route/ground และแปลงเข้า dodeca/goldberg pipeline
  3. ชั้น Metatron/icosa-dodeca logic (runtime routing policy)

  - ใช้ meta_route, meta_cpair, meta_cross เพื่อตัดสินการเดินบน 12-face topology
  - นี่คือส่วน “world AB / face คู่ตรงข้าม / cross ring” ที่คุณถาม
  4. ชั้น Frustum/parallel hook (runtime side path)

  - เขียนข้อมูลเข้า frustum path ควบคู่ dispatch หลัก (ไม่ชน core path)
  - ใช้สำหรับต่อ spatial/frustum zone
  - อ้างอิง: src/tgw_frustum_wire.h, src/geo_compound_cfg.h

  5. ชั้น GeoPixel output/session (runtime output stage)

  - แปลงผลไปเป็น session/timeline GPX4
  - geometry ที่คำนวณก่อนหน้าเป็น input ให้ชั้นนี้
  - อ้างอิง: src/geopixel_session_feed.h, src/geopixel_v21.c

  สรุปสั้น: geometry ของคุณถูกใช้ “ตอนรันจริงทุก packet/address” ตั้งแต่ encode → route → topology decision → output
  ไม่ได้เป็นแค่คอนเซปต์หรือ test-only.
